# Debugging HAMi Scheduler Extender Locally

This document explains how to use [`hack/debug-scheduler-extender-local.sh`](../hack/debug-scheduler-extender-local.sh) to run the Scheduler Extender locally and intercept in-cluster scheduler traffic to your machine with [Telepresence](https://www.telepresence.io/), so you can debug with breakpoints or inspect request logs.

## When to Use This

- HAMi chart is already installed in your Kubernetes cluster (including scheduler Deployment and device ConfigMap).
- You want to run the extender HTTPS server in a local process (for example, in VS Code / GoLand) and receive `/filter` and `/bind` requests from the in-cluster `kube-scheduler`.

## Prerequisites

| Dependency | Notes |
|------|------|
| `kubectl` | Configured and able to access the target cluster. |
| `telepresence` | v2 client. The script runs `telepresence connect` and `telepresence intercept`. |
| `openssl` | Used to generate local self-signed TLS certs when `tls.crt`/`tls.key` do not exist yet. |
| HAMi resources in cluster | A Deployment with label `app.kubernetes.io/component=hami-scheduler` exists, and a ConfigMap named `{deploymentName}-device` exists (chart default naming). |

For Telepresence installation, see [Install Telepresence](https://telepresence.io/docs/quick-start).

## How It Works (High Level)

1. Export device plugin config from the cluster to `.tmp/device-config.yaml`.
2. Generate local TLS certs under `.tmp/scheduler-debug-certs/` (or reuse existing files).
3. Write `.tmp/scheduler-extender-debug.env` for local extender runtime settings.
4. Run `telepresence connect`, then `telepresence intercept` on the scheduler Deployment, mapping pod **remote port** (default `443`) to your **local port** (default `8443`).
5. Start extender locally (for example on `0.0.0.0:8443`) and it receives scheduler traffic.

**Important**: `kube-scheduler` reaches extender via the configured `urlPrefix`. To route traffic to your local process after intercept, set `urlPrefix` to your local HTTPS endpoint (see next section).

## Cluster and Helm Configuration

After intercept, extender container port `443` is forwarded to your local `8443`. Since scheduler accesses extender in the same pod network namespace, you usually want an HTTPS URL with explicit port, for example:

```yaml
scheduler:
  extender:
    urlPrefixOverride: "https://127.0.0.1:8443"
```

Set this in `charts/hami/values.yaml` and upgrade the release. Then scheduler config (`config.yaml` / `config.json` in ConfigMap) uses this `urlPrefix`. When the URL starts with `https://`, TLS settings are enabled accordingly.

If `urlPrefixOverride` is not set, scheduler keeps using the default in-cluster address and will not automatically reach your local debug port.

Additional notes:

- The script writes `HAMI_SCHEDULER_NAME` and related env vars. If you changed `schedulerName` in chart values, make local runtime settings consistent.
- Local certs are self-signed. This works best when scheduler-side extender TLS config allows insecure cert verification (based on current chart behavior).

## Usage

Run from repository root:

```bash
./hack/debug-scheduler-extender-local.sh
```

By default it will:

1. Auto-detect a globally unique scheduler Deployment labeled `app.kubernetes.io/component=hami-scheduler` (if multiple are found, you must pass namespace and deployment explicitly).
2. Generate local cert/env, export device ConfigMap, connect via Telepresence, and create intercept (`8443:443` by default).

### CLI Options

| Option | Description |
|------|------|
| `-n`, `--namespace <ns>` | Namespace of scheduler Deployment. Optional only when auto-detection is unique. |
| `-d`, `--deployment <name>` | Name of scheduler Deployment. Optional only when auto-detection is unique. |
| `--local-port <port>` | Local listening port, default `8443`. Must match `urlPrefixOverride` and local extender listen port. |
| `--remote-port <port>` | Intercepted pod port, default `443` (extender container port). |
| `--cert-dir <path>` | Directory for `tls.crt` / `tls.key`, default `<repo>/.tmp/scheduler-debug-certs`. |
| `--connect-only` | Prepare cert/config/env and connect cluster, but skip `telepresence intercept`. |
| `--leave` | Only run `telepresence leave <deployment>` and exit. |
| `-h`, `--help` | Show help message. |

Example with explicit namespace/deployment and local port `9443`:

```bash
./hack/debug-scheduler-extender-local.sh \
  -n kube-system \
  -d my-release-hami-scheduler \
  --local-port 9443
```

Then set Helm `urlPrefixOverride` to `https://127.0.0.1:9443`, and make local extender listen on `9443`.

### Files Generated by the Script

All files are under repository `.tmp/` (already ignored by `.gitignore`):

| Path | Purpose |
|------|------|
| `.tmp/scheduler-extender-debug.env` | Environment variables to load for local extender runtime. |
| `.tmp/device-config.yaml` | Exported from ConfigMap `{deployment}-device` key `device-config.yaml`. |
| `.tmp/scheduler-debug-certs/tls.crt`, `tls.key` | Local HTTPS cert/key. Reused if already present. |

## Start Extender Locally

1. Run the script until it prints `Ready for local debugging.`.
2. Load `.tmp/scheduler-extender-debug.env` in your IDE/runtime and start extender so it listens on `HAMI_HTTP_BIND` (default `0.0.0.0:8443`).
3. Trigger scheduling in cluster (create/update pods that use HAMi scheduler) and verify local logs receive `/filter` and `/bind` requests.

The VS Code `launch.json` task name shown by the script may vary by repository setup.

## Stop Debugging

```bash
./hack/debug-scheduler-extender-local.sh --leave -n <ns> -d <deployment>
```

If `-n`/`-d` are omitted, the script still attempts auto-detection using the same rules. You can also run `telepresence quit` after finishing (the script already tries `telepresence quit` before each new connect).

## Troubleshooting

**1) Auto-detection failed for Deployment**

No Deployment with label `app.kubernetes.io/component=hami-scheduler`, or multiple were found. Pass `-n` and `-d` explicitly.

**2) Exporting device config failed**

Check ConfigMap name `{deployment}-device` exists and contains key `device-config.yaml`. If you changed chart naming conventions, align them or adjust `cm_name` logic in script.

**3) Scheduler requests are not reaching local machine**

Verify `scheduler.extender.urlPrefixOverride` is set to `https://127.0.0.1:<local-port>` and matches `--local-port` and local process listen port. Also confirm Helm upgrade is applied and scheduler has reloaded the updated ConfigMap.

**4) Telepresence permission or traffic issues**

Confirm your user has namespace permissions and Telepresence version is compatible with the cluster traffic-manager. See Telepresence troubleshooting docs.
